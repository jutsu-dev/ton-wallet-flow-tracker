# TON Wallet Flow Tracker

TON Wallet Flow Tracker загружает публичную ончейн-активность TON-адреса, приводит её к единой модели событий и рисует переводы в виде интерактивного графа. Это приватное веб-приложение с аутентификацией для небольшой команды и одновременно open-source проект под лицензией Apache-2.0.

Приложение работает только с публичными данными блокчейна. Оно никогда не запрашивает seed-фразы, приватные ключи или подключение кошелька, не переводит средства и не берётся доказывать, кому принадлежит адрес.

## Возможности

- **Ввод адреса** — дружественный bounceable (`EQ…`), дружественный non-bounceable (`UQ…`), сырой (`0:…`) и `.ton`-имена. Любой ввод нормализуется в канонический сырой адрес плюс обе дружественные формы и workchain; контрольная сумма проверяется через `@ton/core`.
- **Интерактивный граф переводов** — проверяемый кошелёк в центре, контрагенты вокруг. Рёбра направлены к получателю, несут сумму перевода (TON, символ джеттона, имя NFT или сгруппированный счётчик `×N`), утолщаются с ростом числа операций и становятся пунктирными, когда все сгруппированные переводы неуспешны. Тип узла кодируется стилем рамки, а не только цветом.
- **Раскрытие узлов** — клик по контрагенту подгружает его переводы до настраиваемой глубины (предел по умолчанию — 3). Набор посещённых адресов предотвращает циклы, а жёсткие лимиты 150 узлов / 300 рёбер останавливают неконтролируемый рост с видимым предупреждением.
- **Таблица операций** — каждое нормализованное действие: время, тип, направление, статус, актив, сумма, отправитель, получатель, memo и источник данных. Фильтры по виду актива / направлению / статусу, полнотекстовый поиск по адресам и memo, диапазон дат, сортировка по времени или сумме и постраничный вывод на стороне клиента.
- **Вкладка активов** — балансы джеттонов и принадлежащие NFT, загружаются по запросу.
- **Пользовательские метки** — пометка адреса (`OWN`, `SAFE`, `UNKNOWN`, `SUSPICIOUS`, `SERVICE`, `EXCHANGE`, `MARKETPLACE`, `OTHER`) с названием и заметкой. В каждом представлении метки прямо указано, что метки заданы пользователем и не являются подтверждёнными системой фактами. Изменения меток пишутся в журнал аудита.
- **Экспорт** — диаграмма в PNG, SVG или квадратный PNG 2048×2048 на белом фоне, отрисованный из живого графа.
- **Демо-режим** — при `DEMO_MODE=true` приложение отдаёт только синтетические данные, не обращается к внешним API и показывает постоянный баннер. Удобно для скриншотов, локальных проб и e2e-тестов.

## Архитектура

Четыре намеренно разделённых слоя:

- **Домен** (`src/domain`) — чистые типы и графовая математика без фреймворков (агрегация рёбер, лимиты узлов/рёбер, безопасное к циклам раскрытие, санитизация строк). Ни React, ни ввода-вывода.
- **Провайдеры** (`src/server/providers`) — один интерфейс `BlockchainProvider` с двумя реализациями (TonAPI, TON Center) за оркестратором, обёрнутыми в устойчивый HTTP-клиент (таймаут, ограниченные повторы с джиттером, `Retry-After`, circuit breaker, TTL-кэш, ограничитель конкурентности, allowlist хостов против SSRF).
- **Серверные сервисы** (`src/server`) — анализ, аутентификация, метки, аудит и ограничение частоты поверх Prisma/PostgreSQL.
- **Приложение** (`src/app`, `src/components`) — App Router из Next.js. Серверные компоненты загружают и рендерят данные; несколько клиентских компонентов (граф на React Flow, таблица операций, вкладки меток и активов) отвечают за интерактивность. Обработчики маршрутов в `src/app/api` предоставляют внутренние эндпоинты, а `src/middleware.ts` задаёт nonce для CSP на каждый запрос.

Поток запроса и карта модулей описаны в [ARCHITECTURE.md](ARCHITECTURE.md).

## Стек

Next.js 15 (App Router) и React 19 на TypeScript в строгом режиме, Tailwind CSS для стилей и React Flow (`@xyflow/react`) для графа. Серверное состояние хранится в PostgreSQL через Prisma. Пароли хешируются Argon2id (`@node-rs/argon2`), ввод проверяется Zod и на форме, и на сервере, а адресная математика TON использует `@ton/core`. `@tanstack/react-query` присутствует как зависимость, но приложение опирается на серверные компоненты и небольшой помощник `fetch`, а не на клиентский кэш запросов.

## Быстрый старт (Docker)

Требуется Docker с Compose.

```bash
cp .env.example .env
# Заполните TONAPI_API_KEY / TONCENTER_API_KEY и сгенерируйте секреты ниже.
node -e "console.log(require('crypto').randomBytes(48).toString('base64url'))"   # SESSION_SECRET
node -e "console.log(require('crypto').randomBytes(48).toString('base64url'))"   # AUTH_SECRET
node -e "console.log(require('crypto').randomBytes(24).toString('base64url'))"   # POSTGRES_PASSWORD

docker compose up -d --build
```

Приложение публикуется на `http://127.0.0.1:8137`; PostgreSQL доступен только внутри сети Compose и не выставляется на хост. Миграции и seed выполняются при старте. Seed создаёт первого владельца (`jutsu-dev`) со случайным временным паролем, записанным в `secrets/initial-owner-password`:

```powershell
Get-Content .\secrets\initial-owner-password   # Windows
```
```bash
cat ./secrets/initial-owner-password           # Linux/macOS
```

Войдите с этим паролем; система потребует сменить его перед началом работы. Для TLS поставьте перед приложением обратный прокси. Полная инструкция — в [DEPLOYMENT.md](DEPLOYMENT.md).

Чтобы запустить интерфейс без ключей API, задайте `DEMO_MODE=true` и откройте приложение — оно отдаст встроенный синтетический сценарий.

## Переменные окружения

Вся конфигурация — через переменные окружения. Скопируйте [.env.example](.env.example) в `.env` и заполните; файл документирует каждый ключ: URL и ключи провайдеров, URL базы данных, секреты сессий и аутентификации, лимиты анализа (число анализов в окно, конкурентность, ограничения на исходные события и глубину, лимиты узлов/рёбер), блокировку входа, настройку провайдеров/кэша и публичные Telegram-ссылки проекта (`TELEGRAM_CHANNEL_URL`, `TELEGRAM_ACCESS_URL` — не секреты, имеют рабочие значения по умолчанию). Сервер проверяет окружение через Zod при первом обращении и не стартует, выдавая список проблемных ключей, — при этом их значения он не печатает.

## Аутентификация

Публичной регистрации нет; учётные записи создаёт владелец. Есть две роли: **OWNER** (управляет пользователями и метками, видит журнал аудита) и **MEMBER** (анализирует адреса, смотрит операции, создаёт метки, экспортирует).

Сессии хранятся на сервере в базе данных и адресуются непрозрачным случайным токеном в cookie с флагом `httpOnly`; на сервере хранится только SHA-256 токена. Пароли хешируются Argon2id с базовыми параметрами OWASP. Первый вход по временному паролю принудительно требует смены пароля, что заодно отзывает действующие сессии. Вход защищён блокировкой аккаунта после серии неудач и ограничением частоты по IP, а проверка пароля выполняется даже для несуществующих имён — чтобы затруднить перечисление пользователей по времени ответа.

Первого владельца создаёт seed базы данных со случайным паролем, записанным только в `secrets/initial-owner-password`, — он никогда не попадает в логи или консоль.

## Провайдеры API

TonAPI (`https://tonapi.io`, REST v2) — основной источник; TON Center API v3 (`https://toncenter.com/api/v3`) — резервный. Оркестратор сначала пробует TonAPI и переключается на TON Center только при временных сбоях (ограничение частоты, 5xx апстрима, таймауты, сетевые ошибки, открытый circuit breaker); постоянные ошибки — например, неверный запрос или неверный ключ — пробрасываются, потому что резерв тут не поможет. Каждый результат фиксирует, какой источник его выдал и была ли неполнота ответа.

Ключи API читаются из серверного окружения и используются только в серверном коде (модули `server-only`). Они никогда не отправляются в браузер, не встраиваются в ответы и не пишутся в логи. Запросы ограничены allowlist из двух хостов провайдеров, редиректы отклоняются, схемы кроме HTTPS отвергаются.

## Тесты

`npm test` запускает набор Vitest (юнит-тесты адресов, форматирования сумм, валидации DNS, санитизации, построения графа, маппинга провайдеров, устойчивости, ограничения частоты и CSRF). Интеграционный тест базы данных для аутентификации и сессий завязан на `DATABASE_URL`: он выполняется, когда URL Postgres задан (локально и в CI), и пропускается иначе. `npm run test:e2e` запускает Playwright против работающего инстанса (достаточно демо-режима); браузерные тесты нацеливаются на `E2E_BASE_URL`.

Сначала засейте тестовые учётные записи (`npx tsx e2e/seed-users.ts` с тем же `DATABASE_URL`) и запускайте набор против только что поднятого инстанса. Число входов ограничено по IP клиента (`LOGIN_MAX_ATTEMPTS * 3` за окно блокировки) и хранится в памяти, а все тесты входят с одного адреса — поэтому второй прогон внутри окна исчерпывает лимит и падает на входе. Перезапуск инстанса сбрасывает счётчик, повторный сев — нет. Сев также возвращает одноразовый пароль, который расходует тест принудительной смены пароля.

```bash
npm test            # юнит + (если задан DATABASE_URL) интеграция
npm run typecheck   # tsc --noEmit
npm run lint        # next lint
npm run build       # prisma generate && next build
```

CI не нужны настоящие ключи API: юнит-тесты используют замоканные провайдеры и демо-данные.

## Безопасность

Приложение проверяет каждый серверный ввод через Zod, контролирует аутентификацию и авторизацию на каждом защищённом маршруте, обеспечивает защиту от CSRF двойным токеном плюс проверкой одного источника на мутациях и задаёт строгий Content-Security-Policy с `frame-ancestors 'none'` из middleware. Структурированные JSON-логи вырезают поля, похожие на секреты, и сокращают адреса. Политика безопасности и порядок сообщения о проблемах — в [SECURITY.md](SECURITY.md).

## Ограничения

Резервный путь TON Center классифицирует только TON-переводы; переводы джеттонов и NFT на этом пути помечаются как неполные. Ссылки в эксплореры ведут на адреса, а не на отдельные транзакции, потому что лента событий не несёт хешей отдельных транзакций. Ограничитель частоты работает в пределах процесса. Эти и другие честные оговорки собраны в [LIMITATIONS.md](LIMITATIONS.md).

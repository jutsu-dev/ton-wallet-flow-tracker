# Развёртывание

Здесь описано развёртывание на одном хосте через Docker Compose: приложение и PostgreSQL в общей внутренней сети, база наружу не выставлена, а приложение доступно только на loopback за обратным прокси, который терминирует TLS.

## Требования

- Docker с плагином Compose.
- Обратный прокси (nginx, Caddy, Traefik или аналогичный) для HTTPS перед приложением.
- Ключи API провайдеров: ключ TonAPI и ключ TON Center. Для ознакомления приложение работает и без них — в режиме `DEMO_MODE=true`.

## 1. Настройка окружения

```bash
cp .env.example .env
```

Отредактируйте `.env`:

- `TONAPI_API_KEY`, `TONCENTER_API_KEY` — ключи провайдеров.
- `POSTGRES_DB`, `POSTGRES_USER`, `POSTGRES_PASSWORD` — учётные данные базы. В Docker хост базы — `postgres`, поэтому `DATABASE_URL` выглядит так: `postgresql://USER:PASSWORD@postgres:5432/DB`.
- `SESSION_SECRET`, `AUTH_SECRET` — не менее 32 символов каждый; как их сгенерировать, показано ниже.
- `APP_ENV=production` — при этом значении cookie сессии и CSRF получают флаг `Secure`. В `APP_URL` должен быть указан публичный HTTPS-origin, с которого вы отдаёте приложение: именно с ним сверяется CSRF-проверка совпадения источника.
- `DEMO_MODE=false` для реального развёртывания.

Генерация секретов:

```bash
node -e "console.log(require('crypto').randomBytes(48).toString('base64url'))"   # SESSION_SECRET
node -e "console.log(require('crypto').randomBytes(48).toString('base64url'))"   # AUTH_SECRET
node -e "console.log(require('crypto').randomBytes(24).toString('base64url'))"   # POSTGRES_PASSWORD
```

Держите `.env` вне системы контроля версий (он уже исключён) и ограничьте права на файл пользователем, от имени которого выполняется развёртывание.

> **Смените ключи провайдеров перед любым публичным развёртыванием.** Если ключи хоть раз пересылались в чате, по почте или в тикете, считайте их скомпрометированными: выпустите новые ключи TonAPI и TON Center, впишите новые значения только в локальный `.env`, а старые отзовите. Никогда не коммитьте настоящий ключ и не вставляйте его в issue или PR.

## 2. Сборка и запуск

```bash
docker compose up -d --build
```

Приложение публикуется на `http://127.0.0.1:8137`. Внутри контейнер слушает порт 3000; PostgreSQL доступен только в сети Compose и на хост не публикуется. Миграции базы и seed выполняются при старте, поэтому схема создаётся, а первый владелец заводится автоматически. Данные базы хранятся в постоянном именованном томе.

Проверьте, что всё поднялось:

```bash
docker compose ps
curl -s http://127.0.0.1:8137/api/health
```

Здоровый ответ — `{"status":"ok","database":"up",...}` с HTTP 200; при проблемах с базой возвращается 503.

## 3. Получение пароля первого владельца

Seed создаёт владельца `jutsu-dev` со случайным временным паролём, который записывается только в `secrets/initial-owner-password` (в вывод он никогда не печатается). Прочитать его:

```powershell
Get-Content .\secrets\initial-owner-password    # Windows / PowerShell
```
```bash
cat ./secrets/initial-owner-password             # Linux / macOS
```

Войдите с этим паролем. Сменить его потребуется сразу же; смена отзывает сессию, созданную seed, и открывает новую. После входа и смены пароля файл можно удалить.

## 4. Обратный прокси перед приложением

Приложение слушает loopback и работает по обычному HTTP; TLS оно само не терминирует. Поставьте перед ним обратный прокси, который:

- отдаёт HTTPS и проксирует на `127.0.0.1:8137`;
- передаёт `X-Forwarded-For` (используется для хешированных IP в журнале аудита) и заголовок `Origin` (используется CSRF-проверкой совпадения источника);
- при желании использует `/api/health` как health-check апстрима.

Приложение уже выставляет `X-Content-Type-Options`, `Referrer-Policy`, `X-Frame-Options: DENY`, `Permissions-Policy` и строгий Content-Security-Policy; HSTS должен добавлять ваш прокси.

## Обновление

```bash
git pull
docker compose up -d --build
```

Миграции выполняются при старте и идемпотентны — как и seed (существующего владельца он пропускает). Перед обновлением сделайте резервную копию базы (см. [OPERATIONS.ru.md](OPERATIONS.ru.md)).

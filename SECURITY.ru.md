# Безопасность

Этот документ описывает модель безопасности TON Wallet Flow Tracker и порядок сообщения о
проблеме. Анализ угроз — в [THREAT_MODEL.ru.md](THREAT_MODEL.ru.md), известные пробелы — в
[LIMITATIONS.ru.md](LIMITATIONS.ru.md).

## Сообщение об уязвимости

Откройте приватный security advisory в репозитории на GitHub
(`Security` → `Report a vulnerability`) или обычный issue, если сообщение не содержит
чувствительных деталей. Пожалуйста, не раскрывайте детали эксплуатации публично, пока не выйдет
исправление.

## Что приложение делает и чего не делает

Оно читает публичные ончейн-данные через TonAPI и TON Center. Оно никогда не запрашивает
seed-фразы и приватные ключи, не подключает кошелёк, не подписывает и не отправляет транзакции и
не перемещает средства. Оно не доказывает, кому принадлежит адрес.

## Действующие меры

**Аутентификация и сессии.** Пароли хешируются Argon2id (`@node-rs/argon2`, m=19 MiB, t=2, p=1).
Сессия — непрозрачный случайный токен: в базе хранится только его SHA-256, а ссылается на него
cookie с флагами `httpOnly` и `SameSite=Lax`, которая в продакшене ещё и `Secure`. Публичной
регистрации нет. Серия неудачных входов увеличивает счётчик и блокирует учётную запись на
настраиваемое окно; кроме того, попытки входа ограничены по частоте с одного IP. Отключение
пользователя или смена пароля отзывают все его сессии.

**Авторизация.** Две роли, `OWNER` и `MEMBER`, проверяются на сервере. Операции только для
владельца (управление пользователями, удаление любой метки) участникам недоступны.

**CSRF.** Изменяющие запросы требуют двойной отправки токена: cookie `twft_csrf` без флага
`httpOnly`, значение которой нужно продублировать в заголовке `x-csrf-token`, плюс проверка
совпадения источника по заголовку `Origin`. Вход, у которого токена ещё нет, защищён проверкой
совпадения источника.

**Content Security Policy.** `src/middleware.ts` выставляет nonce на каждый запрос и CSP с
`default-src 'self'`, `frame-ancestors 'none'`, `object-src 'none'`; скрипты ограничены тем же
источником плюс nonce запроса. Дополнительные заголовки (`X-Content-Type-Options`,
`Referrer-Policy`, `X-Frame-Options: DENY`, `Permissions-Policy`) задаются в `next.config.mjs`.

**SSRF и исходящие запросы.** HTTP-клиент сверяет хост каждого апстрим-URL с allowlist,
построенным из настроенных базовых URL провайдеров, отклоняет схемы, отличные от http(s), и
использует `redirect: 'error'`. Загрузки произвольного URL и прокси изображений нет; список
remote patterns у `next/image` пуст.

**Проверка ввода.** Каждый обработчик маршрута проверяет тело или строку запроса через Zod до
использования. Prisma параметризует все запросы; SQL, собранного из строк, нет, как нет ни
`eval`, ни запуска shell-команд из пользовательского ввода.

**Недоверенные метаданные.** Названия токенов и NFT, комментарии, memo и внешние URL считаются
недоверенными: управляющие символы вырезаются, длина ограничивается, URL допускаются только
http(s), и ничто не рендерится как сырой HTML (React экранирует при выводе).

**Секреты.** Ключи API и учётные данные базы остаются на сервере. Они никогда не уходят клиенту и
не попадают в `NEXT_PUBLIC_*`, а `.env` исключён из git, и права на файл ограничены.
Структурированные логи маскируют поля, похожие на секреты, и сокращают адреса.

**Лимиты частоты и размера.** Анализы ограничены по частоте для каждого пользователя, запросы к
апстримам ограничены таймаутом и прикрыты circuit breaker и ограничителем конкурентности, а граф
ограничен 150 узлами / 300 рёбрами при глубине раскрытия ≤ 3.

## Сканирование зависимостей и секретов

CI запускает `npm audit --omit=dev --audit-level=high`, `gitleaks` по всей истории,
продакшен-сборку, сборку Docker-образа и проверки документации. Аудит намеренно охватывает только
продакшен-зависимости и падает на уровне high и critical: это та поверхность, до которой может
дотянуться пользователь развёрнутого приложения. Два предупреждения средней серьёзности приходят
из версии postcss, встроенной в Next.js, — они ниже этого порога и находятся в инструментарии
разработки, поэтому CI на них по замыслу не падает; этот код обрабатывает только собственный CSS
проекта на этапе сборки и недостижим в рантайме. Чтобы увидеть их, запустите `npm audit` без
флагов. См. LIMITATIONS.ru.md.

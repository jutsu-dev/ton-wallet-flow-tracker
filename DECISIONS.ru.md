# DECISIONS

Текущий журнал решений, принятых по ходу работы над проектом, вместе с обоснованием и
компромиссами. Новые записи добавляются в конец соответствующего раздела.

## Окружение

**Хост — Windows 11, а не Linux VPS.** Техническое задание исходило из Linux VPS (`/opt`,
`chmod 600`, `sudo cat`). Фактический хост — Windows 11 Pro с PowerShell. Адаптации:

- Корень проекта — существующая рабочая директория
  `…\Проекты\tonwallet`, а не `/opt/ton-wallet-flow-tracker`. Директория была пуста, и именно в
  ней была открыта сессия; переносить её было бы неожиданно и ничего не даёт.
- Ужесточение прав на файлы делается через NTFS ACL и `icacls`, а не через `chmod 600`. Доступ
  к `.env` и `secrets/` ограничен текущим пользователем, наследование прав отключено. Это
  ближайший в Windows аналог режима `600` «только владелец».
- Команда «покажи пароль владельца» приведена в документации в Windows-форме
  `Get-Content .\secrets\initial-owner-password` (без `sudo`).
- Папка проекта находится внутри пути, синхронизируемого с OneDrive. Чтобы избежать лишней
  синхронизации и блокировок файлов, всё тяжёлое (`node_modules`, `.next`, результаты сборки,
  данные БД) исключено из git, а сама база живёт в именованном томе Docker в его собственном
  хранилище, а не в синхронизируемой папке.

**Порт приложения: `127.0.0.1:8137`.** Порты 3000/5432/8137 все были свободны. Приложение
публикуется только на `127.0.0.1:8137`; внутри контейнер слушает 3000. PostgreSQL не
публикуется вообще.

**Авторизация GitHub CLI задерживала публикацию.** Большую часть работы `gh auth status` не
сообщал ни об одном хосте, поэтому всё, что предшествует шлагбауму публикации, собиралось
локально, а публикация ждала. Владелец с тех пор вошёл как `jutsu-dev`, и репозиторий
опубликован.

**Демон Docker на старте не работал.** Docker Desktop установлен, но его движок не был запущен
во время первичной инвентаризации. Работа без Docker (линт, проверка типов, юнит-тесты,
продакшен-сборка) в нём не нуждается; демон запускается перед этапами контейнерного
развёртывания и интеграции с БД.

## Атрибуция и git

- Локальный для репозитория автор git — `jutsu-dev <jutsu-dev@users.noreply.github.com>`,
  задан только через `git config --local`. Глобальный конфиг git не тронут. Адрес noreply не
  раскрывает личную почту и при этом ведёт на аккаунт. Как только появится `gh auth`, его можно
  будет заменить на числовую форму `ID+jutsu-dev@users.noreply.github.com`.
- Никаких трейлеров с соавторами и никакой атрибуции автоматических инструментов — ни в
  истории, ни в документации.

## Секреты

- `.env` генерируется локально одноразовым скриптом, который лежит в черновой папке вне
  репозитория и удаляется сразу после использования, поэтому два временных ключа API не
  попадают ни в git, ни в историю оболочки, ни в этот отчёт. `POSTGRES_PASSWORD`,
  `SESSION_SECRET`, `AUTH_SECRET` получены из `crypto.randomBytes` (24 / 48 / 48 байт,
  base64url).
- Два переданных ключа TonAPI / TON Center считаются скомпрометированными (их передали в
  приватном канале), и изначально их ротация была условием публикации. Владелец это условие
  снял: ключи существуют только в локальном `.env`, который игнорируется git, и не попадают ни в
  git, ни в сборку, ни в опубликованные артефакты, что подтверждает сканирование секретов, —
  поэтому публикация прошла без ротации. Ротация остаётся рекомендованным следующим шагом;
  новые ключи попадают только в локальный `.env`, и в репозитории при этом ничего не меняется.
  См. `PUBLICATION_CHECKLIST.ru.md`.

## Выбор стека

- **Argon2id** для хеширования паролей (первый выбор по спецификации; memory-hard, актуальная
  рекомендация OWASP). Используется нативная привязка `argon2`.
- **Prisma + PostgreSQL** для хранения; перечисления смоделированы как enum-ы Prisma.
- **Сессии в базе данных**, адресуемые непрозрачным случайным значением cookie, вместо JWT.
  Серверные сессии проще отзывать (отключение аккаунта должно срабатывать немедленно), и они не
  хранят секрет в самом токене.
- **TanStack Query** для загрузки данных и кэша на клиенте; **Zod** для валидации, общей для
  клиентских форм и серверных обработчиков маршрутов.
- **React Flow** для графа: он берёт на себя панорамирование и зум, кастомные узлы и рёбра и
  даёт удобный для экспорта DOM/SVG.

## Стратегия провайдеров

- Один интерфейс `BlockchainProvider`, две реализации и оркестратор, который сначала пробует
  TonAPI, а затем TON Center. Таймаут 10 с, ≤ 3 попыток, экспоненциальная задержка + джиттер,
  соблюдение `Retry-After`, circuit breaker на каждого провайдера, небольшой TTL-кэш и
  ограничитель конкурентности.
- Отсутствующие поля становятся `null` и помечаются флагом `isIncomplete`; ничего не
  выдумывается. Каждый ответ помечается своим источником, чтобы интерфейс мог показать
  происхождение данных.

## Развёртывание и сборка

- **Prisma `binaryTargets = ["native", "debian-openssl-3.0.x"]`.** В `node:22-bookworm-slim`
  нет пакета openssl, поэтому `prisma generate` ошибочно определял движок как
  `debian-openssl-1.1.x`, тогда как в рантайме (где openssl установлен) версия 3.0. Фиксация
  Debian-таргета и установка openssl на стадии сборки устранили рассогласование движков.
- **Отказ от `output: 'standalone'`.** Docker-рантайм копирует полный `node_modules` вместе с
  `.next` и запускает `next start` — с Prisma это проще, чем трассировать standalone-сервер.
  Standalone добавлял лишь неиспользуемую директорию и предупреждение при старте.
- **Одноразовый dev-PostgreSQL** (`twft-devdb`, `127.0.0.1:5433`, пароль только для разработки)
  использовался, чтобы написать миграции и прогнать интеграционные тесты БД и Playwright, — так
  продакшен-пароль из `.env` ни разу не понадобился за пределами контейнера.
- **Свой структурированный логгер** вместо pino, чтобы не тащить в бандл Next воркер-треды
  транспортов pino и сохранить полный контроль над вырезанием секретов и хостов.

## Корректировки объёма работ

- **Админ-страница владельца** (`/admin`) выводит список пользователей, создаёт учётные записи,
  переключает активность и роль и показывает журнал аудита — этого достаточно для управления
  пользователями со стороны владельца и для теста «MEMBER не может попасть на страницы OWNER».
  Она намеренно минимальна.
- **Скриншоты** снимаются opt-in-спекой Playwright в демо-режиме, поэтому каждый пиксель на них
  синтетический, а пересоздание — это одна команда.
- **Режим трассировки актива** поставляется как возможность провайдера (эндпоинты истории NFT),
  но пока без отдельного экрана в интерфейсе; он числится в ROADMAP, а не заявлен как готовая
  функция.

# Участие в разработке

Спасибо за интерес к развитию TON Wallet Flow Tracker. Здесь описаны локальная настройка, проверки, которые должно пройти изменение, и соглашения, принятые в проекте.

Участвуя в разработке, вы соглашаетесь, что ваша работа распространяется по лицензии проекта [Apache-2.0](LICENSE) и что вы соблюдаете [Кодекс поведения](CODE_OF_CONDUCT.ru.md).

## Настройка окружения

Нужны Node.js 20+ и база данных PostgreSQL. Проще всего поднять её через Docker.

```bash
git clone https://github.com/jutsu-dev/ton-wallet-flow-tracker.git
cd ton-wallet-flow-tracker
npm install
cp .env.example .env
```

Заполните `.env`. Для большей части задач настоящие ключи провайдеров не нужны — поставьте `DEMO_MODE=true`, и приложение будет отдавать синтетические фикстуры, не обращаясь ни к одному внешнему API. Понадобятся `DATABASE_URL`, `SESSION_SECRET` и `AUTH_SECRET` (последние два — не менее 32 символов каждый); секреты генерируются так:

```bash
node -e "console.log(require('crypto').randomBytes(48).toString('base64url'))"
```

### С Docker

Стек Compose поднимает приложение и PostgreSQL вместе, выполняет при старте миграции и seed и публикует приложение на `127.0.0.1:8137`:

```bash
docker compose up -d --build
```

### Без Docker

Направьте `DATABASE_URL` на запущенный PostgreSQL, затем:

```bash
npm run prisma:migrate:dev   # create the schema
npm run db:seed              # create the first owner (password in secrets/initial-owner-password)
npm run dev                  # http://localhost:3000
```

## Обязательные проверки

Каждый pull request должен быть зелёным по всем пяти. Прогоните их локально перед push:

```bash
npm run lint         # next lint
npm run typecheck    # tsc --noEmit (strict)
npm test             # vitest
npm run build        # prisma generate && next build
npm run check:docs   # ссылки в Markdown + русские версии
```

CI проверяет больше, чем этот список: он также запускает `npx prisma validate`, собирает
Docker-образ, выполняет `npm audit --omit=dev --audit-level=high` и сканирует всю историю git
через gitleaks. Локально для них не нужно ничего, кроме Docker, но PR, который их ломает,
станет красным.

Примечания:

- **Тесты.** `npm test` прогоняет юнит-набор с мок-провайдерами и демо-фикстурами, поэтому ключи API не нужны — так же он запускается и в CI. Интеграционный тест аутентификации и сессий завязан на `DATABASE_URL`: он выполняется, когда URL Postgres задан (локально и в CI, где поднимается сервис Postgres), и пропускается в остальных случаях. Пожалуйста, покрывайте новую логику тестами; слои domain и providers чистые, их легко тестировать напрямую.
- **Типы.** Проект на TypeScript в строгом режиме с `noUncheckedIndexedAccess`. Избегайте `any`; недоверенный ввод лучше сужать существующими хелперами `parse` и схемами Zod.
- **Форматирование.** `npm run format` применяет Prettier (с плагином сортировки классов Tailwind). `npm run format:check` проверяет результат.
- **Сквозные тесты.** `npm run test:e2e` запускает Playwright против запущенного экземпляра (демо-режима достаточно). Укажите свой экземпляр через `E2E_BASE_URL`. Сначала засейте тестовые учётные записи и поднимите **свежий** экземпляр: число входов ограничено по IP клиента и хранится в памяти, поэтому второй прогон внутри окна блокировки падает на входе. См. [TUTORIAL.ru.md](TUTORIAL.ru.md#b-локальная-разработка-без-docker).
- **Документация.** У каждого публичного Markdown-документа должна быть русская версия `FILE.ru.md`, обновлённая в том же коммите. `npm run check:ru` это проверяет, и без неё CI падает. `npm run check:links` проверяет внутренние ссылки, изображения и якоря; внешние ссылки проверяются только по запросу через `npm run check:links:external`, поэтому CI не зависит от доступности сторонних сайтов.

## Ветки и коммиты

- Работайте в тематической ветке от `main` — например, `feat/asset-trace`, `fix/toncenter-offset` или `docs/api-examples`.
- Держите коммиты небольшими и сфокусированными и используйте [Conventional Commits](https://www.conventionalcommits.org/): `feat:`, `fix:`, `docs:`, `refactor:`, `test:`, `chore:` и так далее. Область указывать приветствуется, например `fix(providers): honor Retry-After on 429`.
- Пишите сообщения коммитов обычным текстом. Никаких эмодзи в заголовках коммитов и названиях PR.
- Перед открытием PR сделайте rebase на актуальный `main`, чтобы история оставалась линейной и удобной для ревью.

## Pull request

Открывайте PR в `main` и заполняйте шаблон. Хороший PR:

- объясняет, что изменилось и зачем, и ссылается на связанную задачу;
- проходит lint, typecheck, тесты и сборку;
- добавляет или обновляет тесты под изменившееся поведение;
- обновляет документацию, когда меняются поведение, конфигурация или состав API;
- не содержит секретов, ключей API, реальных расследуемых адресов и хостнеймов инфраструктуры.

## Границы и архитектура

Сохраняйте разделение слоёв: чистая логика — в `src/domain`, преобразование данных конкретного провайдера — в `src/server/providers`, оркестрация — в `src/server`. Новые обращения к апстримам идут через интерфейс `BlockchainProvider` и устойчивый рантайм — не вызывайте `fetch` к сторонним хостам напрямую и не добавляйте хосты вне allowlist провайдеров без обсуждения. Если сомневаетесь насчёт крупного изменения, сначала откройте issue, чтобы договориться о подходе.

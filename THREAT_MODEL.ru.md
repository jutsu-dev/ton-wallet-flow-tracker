# Модель угроз

Краткая практическая модель угроз. Она не претендует на полноту: здесь зафиксированы активы,
которые стоит защищать, границы доверия, рассмотренные угрозы и остаточные риски.

## Область и допущения

- Приложение развёрнуто за обратным прокси, который терминирует TLS. Само приложение слушает
  `127.0.0.1` и никогда не выставляется в интернет напрямую.
- Пользователи — небольшая доверенная команда. Учётные записи создаёт владелец; публичной
  регистрации нет.
- Все данные о кошельках — публичные ончейн-данные, полученные из TonAPI и TON Center.
  Приложение не хранит ончейн-активы и не распоряжается ими.

## Активы

1. Учётные данные пользователей и сессии.
2. Ключи API провайдеров и пароль базы данных.
3. База данных: пользователи, кошельки, пользовательские метки, проверки и журнал аудита.
4. Приватные аналитические заметки (метки и заметки к ним видны только вошедшим пользователям).
5. Хост, на котором работает приложение.

## Границы доверия

- **Браузер ↔ приложение.** Здесь проходит недоверенный ввод клиента. Меры: аутентификация по
  сессии, CSRF (двойная отправка токена + проверка совпадения источника), проверка через Zod, CSP
  и защищённые cookie.
- **Приложение ↔ провайдеры TON.** Ответы апстримов — недоверенные данные. Меры: allowlist хостов
  против SSRF, таймауты, повторы с circuit breaker, защитный разбор JSON (отсутствующие поля
  становятся `null` и помечаются как неполные, а не выдумываются) и санитизация метаданных.
- **Приложение ↔ база данных.** Меры: параметризация запросов Prisma и минимальная открытость
  (PostgreSQL не публикуется за пределы внутренней сети Docker).

## Угрозы и меры

| Угроза | Меры |
|---|---|
| Подбор учётных данных / перебор | Argon2id, блокировка учётной записи после серии неудач, ограничение частоты по IP, общие сообщения об ошибках |
| Кража / фиксация сессии | Непрозрачные токены, хранимые в виде хеша, cookie с `httpOnly` + `Secure` + `SameSite=Lax`, отзыв всех сессий при смене пароля / отключении |
| CSRF на изменяющих маршрутах | Двойная отправка `x-csrf-token` + проверка `Origin` |
| Повышение привилегий (MEMBER → OWNER) | Серверная проверка роли на каждом действии, доступном только владельцу |
| XSS через ончейн-метаданные | Экранирование вывода в React, nonce в CSP, санитизация названий/комментариев/memo, никакого сырого HTML |
| SQL-инъекция | Только параметризованные запросы Prisma |
| SSRF через ввод адреса/DNS | Allowlist хостов апстримов, только http(s), `redirect: 'error'`, никаких произвольных запросов и прокси изображений |
| Отказ в обслуживании | Лимиты частоты анализов, таймауты апстримов + circuit breaker, ограничитель конкурентности, лимиты узлов и рёбер, ограничение размера тела запроса |
| Утечка секретов | Ключи только на сервере, `.env` исключён из git + права ограничены, маскирование в логах, никаких секретов в клиентском бандле |
| Отказ от авторства действий | Журнал аудита для входов, изменений меток, управления пользователями и анализов (с хешированным IP) |
| Утечка стектрейсов / заголовков апстрима | Ошибки провайдеров классифицируются в безопасные коды; стектрейсы апстримов и заголовки авторизации до клиента не доходят |

## Остаточные риски

- Ограничитель частоты и circuit breaker живут в памяти процесса. Многоинстансному развёртыванию
  понадобилось бы общее хранилище; проект рассчитан на один экземпляр.
- Достоверность данных зависит от TonAPI и TON Center. Приложение может сообщить только то, что
  они вернули, и помечает частичные ответы как неполные.
- TLS отдан обратному прокси. Если прокси настроен неверно, `Secure`-cookie и модель проверки
  источника слабеют.
- Два ключа API, выданные при первоначальной настройке, передавались по приватному каналу, поэтому
  их следует считать известными третьей стороне. Изначально их ротация была условием публикации;
  вместо этого владелец принял остаточный риск, поскольку ключи существуют только в локальном
  `.env`, который игнорируется git, и не встречаются ни в одном коммите, ни в сборке, ни в
  опубликованных артефактах. Ротация остаётся рекомендованным следующим шагом.
  См. PUBLICATION_CHECKLIST.ru.md.
- Предупреждение postcss на этапе сборки (встроенного в Next.js) принято как допустимое, потому
  что он обрабатывает только собственный CSS проекта во время сборки.

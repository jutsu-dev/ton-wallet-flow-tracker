# Архитектура

Кодовая база устроена так, чтобы чистая логика никогда не зависела от фреймворков, детали провайдеров не просачивались за пределы слоя провайдеров, а Next.js-приложение общалось с доменом только через серверные сервисы. Этот документ описывает слои и разбирает по шагам, что происходит при анализе адреса.

## Слои

```
src/
  domain/        pure types + graph math + sanitization (no React, no I/O)
  lib/           env, validation, logging, TON address/format/DNS, small helpers
  server/
    providers/   BlockchainProvider interface, TonAPI + TON Center, resilient runtime
    analysis/    analyzeWallet / expandNode, graph DTO builder, assets, dashboard
    auth/        password, session, CSRF, web glue, core auth service
    labels/      user label CRUD
    db.ts, audit.ts, rate-limit.ts, http.ts
  app/           Next.js App Router: pages, layouts, route handlers
  components/    client components (React Flow graph, tables, tabs, UI kit)
  middleware.ts  per-request CSP nonce + response headers
prisma/          schema, migration, seed
```

### Домен (`src/domain`)

Без фреймворков и детерминирован. `types.ts` задаёт нормализованную модель (`WalletAction`, `AccountSummary`, `JettonBalance`, `NftItemSummary`), общую для всего, что лежит выше. `graph.ts` — движок графа: `directionRelativeTo` классифицирует действие относительно центрального адреса, `buildEdges` агрегирует действия в направленные рёбра с ключом `(sender, recipient, asset)` и складывает суммы взаимозаменяемых активов через `BigInt`, `enforceGraphLimits` подрезает граф до лимитов по узлам/рёбрам, ранжируя узлы по весу инцидентных рёбер и всегда сохраняя центр, а `selectExpansionTargets` выбирает безопасные к циклам цели раскрытия с учётом набора посещённых. `sanitize.ts` чистит недоверенные строки метаданных (убрать управляющие символы, схлопнуть пробелы, ограничить длину, удалить теговые последовательности) и пропускает только URL со схемой `http(s)`. Этот слой покрыт юнит-тестами напрямую и не выполняет ввода-вывода.

### Провайдеры (`src/server/providers`)

Один интерфейс, две реализации, оркестратор и стек устойчивости.

- `types.ts` объявляет интерфейс `BlockchainProvider` (`validateAddress`, `normalizeAddress`, `resolveDns`, `getAccount`, `getAccountEvents`, `getTransactions`, `getJettonBalances`, `getNftItems`, `getNftHistory`, `getTrace`, `getTransaction`), конверт `ProviderResult<T>` (`data`, `source`, `incomplete`, `warnings`) и `ProviderError`, который классифицирует сбои и помечает, какие из них временные. Ошибки никогда не несут стектрейсы апстрима или заголовки авторизации.
- `tonapi.ts` отображает формы событий и действий TonAPI в `WalletAction` (переводы TON, переводы джеттонов, переводы и покупки NFT, вызовы контрактов). `toncenter.ts` отображает сырые сообщения транзакций v3 в действия-переводы TON и помечает их неполными, потому что резервный провайдер не способен восстановить семантику джеттонов и NFT. `parse.ts` содержит защитные аксессоры, которые никогда не бросают исключение на некорректном JSON.
- `http.ts` — устойчивый fetch: allowlist хостов против SSRF, таймаут на каждый запрос через `AbortController`, ограниченные повторы с экспоненциальной задержкой и джиттером, обработка `Retry-After`, `redirect: 'error'` и классификация статуса в ошибку. `circuit-breaker.ts`, `cache.ts` (TTL-кэш плюс ограничитель конкурентности) и `runtime.ts` связывают всё это вместе: попадание в кэш минует сеть, открытый breaker падает сразу, а ограничитель держит в рамках число запросов в полёте.
- `index.ts` предоставляет `ProviderOrchestrator`: он пробует основной провайдер (TonAPI) и переключается на резервный (TON Center) только тогда, когда основной бросает временную ошибку. `getProvider()` собирает настроенный из окружения экземпляр один раз на процесс.

### Серверные сервисы (`src/server`)

- **analysis** — `service.ts` оркестрирует проверку: провалидировать, при необходимости разрешить `.ton` через DNS, нормализовать адрес, применить пользовательский rate limit, загрузить аккаунт и события, сохранить `Wallet` через upsert и строку `WalletCheck` (плюс запись аудита), подтянуть метки для адресов в поле зрения и построить DTO графа. `graph-builder.ts` превращает агрегированные рёбра и адреса узлов в формы `GraphNodeDto` / `GraphEdgeDto`, которые рендерит клиент, попутно считая для каждого узла число входящих и исходящих операций и сопоставляя тип метки виду узла. `assets.ts` и `dashboard.ts` обслуживают вкладку активов и представления дашборда и истории.
- **auth** — `password.ts` (хеширование и проверка Argon2id), `session.ts` (создание, поиск и уничтожение сессий; токены хранятся в виде хешей), `csrf.ts` (сравнение токенов двойной отправки и проверка одного источника), `web.ts` (обвязка вокруг `cookies()`/`headers()` из Next, `getCurrentUser`, `requireUser`, `requireOwner`, `verifyCsrf`) и `service.ts` (проверка учётных данных с блокировкой и ограничением частоты, смена пароля, функции управления пользователями). Основной сервис намеренно обходится без `next/headers`, чтобы его можно было тестировать против базы данных вне контекста запроса.
- **labels** — создание, перечисление и удаление пользовательских меток с проверкой прав: участник может удалить только свои, владелец — любые.
- **сквозные модули** — `db.ts` (один клиент Prisma на процесс), `audit.ts` (append-only записи аудита; IP хешируются секретом аутентификации и никогда не хранятся в сыром виде), `rate-limit.ts` (корзины с фиксированным окном в памяти) и `http.ts` (помощник для JSON-ошибок, чтение тела, страж CSRF, сопоставление кода ошибки со статусом).

### Next.js App Router (`src/app`)

Всю работу с данными делают серверные компоненты: `(app)/layout.tsx` закрывает доступ, перенаправляет пользователей, обязанных сменить пароль, и показывает демо-баннер; `(app)/page.tsx` рендерит дашборд; `(app)/wallet/[address]/page.tsx` выполняет анализ на сервере и передаёт результат клиентскому `WalletView`. Обработчики маршрутов в `app/api` предоставляют внутренние эндпоинты (см. [API.ru.md](API.ru.md)). `middleware.ts` генерирует nonce на каждый запрос и задаёт строгий Content-Security-Policy (`default-src 'self'`, `frame-ancestors 'none'`, `object-src 'none'`, `script-src` на основе nonce), а `next.config.mjs` добавляет статические заголовки безопасности.

### Клиентские компоненты (`src/components`)

`wallet/flow-graph.tsx` — граф на React Flow: радиальная начальная раскладка, кастомные узлы, у которых стиль рамки кодирует вид, рёбра, чья толщина растёт с числом операций, а пунктир отмечает неуспех, раскрытие узла через `POST /api/expand` и экспорт в PNG/SVG/квадрат через `html-to-image`. `wallet/operations-table.tsx` фильтрует, ищет, сортирует и разбивает на страницы загруженные действия на стороне клиента. Вкладки меток и активов ходят за данными через небольшой типизированный клиент (`src/lib/client/api.ts`), который подставляет заголовок CSRF. Данные текут в одну сторону: серверные компоненты загружают их и передают вниз через пропсы, клиентские компоненты вносят изменения через API-маршруты.

### Хранение (Prisma / PostgreSQL)

Схема (`prisma/schema.prisma`) описывает пользователей, сессии, кошельки, метки кошельков, проверки кошельков, события кошельков и журналы аудита, с перечислениями для роли, типа метки и статуса проверки. Суммы хранятся строками, чтобы сохранить точность. Seed создаёт первого владельца идемпотентно.

## Поток запроса: анализ адреса

1. Аналитик отправляет форму с адресом. Браузер переходит на `/wallet/<address>?limit=&depth=` — маршрут серверного компонента.
2. `wallet/[address]/page.tsx` читает текущего пользователя из cookie сессии, заново проверяет `limit` и `depth` общей Zod-схемой и вызывает `analyzeWallet({ address, limit, depth }, user)`.
3. Если включён `DEMO_MODE`, `analyzeWallet` сразу отдаёт синтетические фикстуры. Иначе он применяет пользовательский rate limit, разрешает ввод (`.ton`-имя — через провайдера, всё остальное нормализуется через `@ton/core`) и зажимает `limit`/`depth` до настроенных максимумов.
4. Он вызывает `getProvider().getAccount()` и `getAccountEvents()`. Оркестратор пробует TonAPI; при временном сбое повторяет запрос с нарастающей задержкой, а если и это не помогает — переключается на TON Center. Каждый ответ несёт свой источник и флаг неполноты.
5. Ответы провайдеров уже нормализованы в `WalletAction`. Сервис делает upsert `Wallet`, записывает `WalletCheck` (статус `SUCCESS` или `PARTIAL`) и запись аудита — сбои сохранения проглатываются, чтобы они не могли сломать ответ, — и подгружает метки для адресов, которые появятся в виде узлов.
6. `buildGraphDto` агрегирует действия в рёбра, применяет лимиты по узлам и рёбрам (выставляя `truncated`, если пришлось подрезать) и выдаёт DTO узлов и рёбер.
7. Страница рендерит `WalletView` со сводкой по аккаунту, графом, действиями, метками и историей. Дальше за панорамирование, зум, раскрытие и экспорт отвечает клиентский `FlowGraph`; раскрытие узла отправляет `POST /api/expand`, который прогоняет тот же конвейер для этого адреса и вливает вернувшиеся узлы и рёбра в живой граф с соблюдением тех же лимитов.

На всём протяжении ключи провайдеров остаются в серверных модулях, недоверенные строки санитизируются перед показом, а логи вырезают секреты и сокращают адреса.

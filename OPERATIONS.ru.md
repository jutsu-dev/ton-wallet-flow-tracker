# Эксплуатация

Повседневная работа с запущенным экземпляром: состояние, логи, резервные копии, пользователи, ротация секретов и параметры, управляющие лимитами.

## Состояние

`GET /api/health` сообщает о живости приложения и доступности базы без аутентификации:

```json
{ "status": "ok", "service": "ton-wallet-flow-tracker", "database": "up", "time": "..." }
```

Он возвращает HTTP 200, когда база отвечает на тривиальный запрос, и HTTP 503 с `"database": "down"`, когда не отвечает, — так что health-check контейнера или прокси может перезапустить сломавшийся экземпляр. Ответ отдаётся с `Cache-Control: no-store`.

```bash
curl -si http://127.0.0.1:8137/api/health | head -n 1
```

## Логи

Логи — структурированный JSON, по одному объекту на строку, в stdout/stderr. Читайте их через `docker compose logs -f` и отправляйте туда, где вы централизуете логи. Эксплуатационно важны два свойства:

- **Секреты вырезаются.** Любое поле, ключ которого выглядит чувствительным (authorization, cookie, api-key, token, password, secret, session, bearer), заменяется на `[redacted]` — рекурсивно, включая вложенные объекты.
- **Адреса сокращаются.** Хелперы логирования выводят сокращённую форму `EQAb…wxyz`, поэтому полные адреса в системных логах не оседают. PID и имени хоста в записях логов нет.

Уровень подробности задаётся переменной `LOG_LEVEL` (`debug`, `info`, `warn`, `error`; по умолчанию `info`).

Отдельный журнал аудита пишется в таблицу `audit_logs` для действий, значимых с точки зрения безопасности: входы, блокировки, смены паролей, изменения пользователей, создание и удаление меток, анализы. IP-адреса там хранятся только как хеш с солью (на основе `AUTH_SECRET`) и никогда в открытом виде.

## Резервные копии

Регулярно и перед каждым обновлением делайте резервную копию тома PostgreSQL. Вспомогательные скрипты оборачивают `pg_dump`/`pg_restore` для базы в Compose:

```bash
scripts/backup-db.sh                 # write a timestamped dump
scripts/restore-db.sh <dump-file>    # restore from a dump
scripts/verify-backup.sh <dump-file> # restore into a throwaway database and sanity-check
```

Храните дампы вне хоста и проверяйте их: резервная копия, которую вы ни разу не восстанавливали, — это надежда, а не резервная копия. Шаг `verify` восстанавливает дамп во временную базу, так что убедиться в его пригодности можно, не трогая продакшен. Если вы вместо этого копируете сам именованный том, снимайте снапшот при остановленной базе либо используйте `pg_dump` для согласованной логической копии.

## Пользователи

Первого владельца создаёт seed (`npm run db:seed` / при старте контейнера) с временным паролем в `secrets/initial-owner-password`; первый вход принудительно требует смены пароля.

Остальные учётные записи заводит владелец на **странице `/admin`** — она доступна из шапки при входе с ролью OWNER. Там можно посмотреть список пользователей, создать учётную запись, переключить активность и роль, а также увидеть последние записи журнала аудита. За страницей стоят сервисные функции, доступные только владельцу (`createUser`, `setUserRole`, `setUserActive`, `listUsers` в `src/server/auth/service.ts`), подключённые через обработчики маршрутов в `src/app/api/admin/users`; оба слоя покрыты тестами.

Новые пользователи создаются с выставленным `mustChangePassword`, поэтому при первом входе они обязаны задать собственный пароль. Передавайте временный пароль по отдельному каналу. Отключение пользователя (`setUserActive(false)`) немедленно отзывает его сессии. Ушедших сотрудников деактивируйте, а не удаляйте, чтобы ссылки в аудите остались целыми.

## Ротация секретов

- **`SESSION_SECRET` / `AUTH_SECRET`.** Обновите значения в `.env` и перезапустите. Ротация `AUTH_SECRET` меняет соль для хеширования IP, поэтому исторические хешированные IP в журнале аудита перестанут соотноситься с новыми — это ожидаемо. Существующие сессии переживают перезапуск, потому что сам токен сессии случаен и хранится в виде хеша, независимо от этих секретов.
- **`POSTGRES_PASSWORD`.** Меняйте его в PostgreSQL и в `DATABASE_URL` одновременно, затем перезапустите приложение.
- **Ключи API провайдеров.** Выпустите новые ключи TonAPI / TON Center, замените их в `.env` и перезапустите. Старые ключи отзовите на стороне провайдера. Если ключ хоть раз покидал сервер, меняйте немедленно.

Чтобы заставить всех пользователей пройти аутентификацию заново, очистите таблицу `sessions`.

## Лимиты и настройка

Лимиты анализа и защитные лимиты задаются переменными окружения (описаны в `.env.example`). Основные:

| Переменная | Значение | По умолчанию |
|---|---|---|
| `MEMBER_ANALYSES_PER_WINDOW` | Анализов на участника за окно (владельцам — 3×) | 10 |
| `MEMBER_ANALYSIS_WINDOW_MINUTES` | Длительность этого окна | 10 |
| `MEMBER_MAX_CONCURRENT_ANALYSES` | Предел конкурентности (заодно задаёт размер ограничителя провайдера) | 3 |
| `MAX_SOURCE_EVENTS` | Максимум исходных событий, загружаемых за один анализ | 100 |
| `MAX_EXPANSION_DEPTH` | Максимальная глубина раскрытия узлов | 3 |
| `MAX_GRAPH_NODES` / `MAX_GRAPH_EDGES` | Жёсткие лимиты графа | 150 / 300 |
| `LOGIN_MAX_ATTEMPTS` | Неудачных входов до блокировки аккаунта | 5 |
| `LOGIN_LOCKOUT_MINUTES` | Длительность блокировки | 15 |
| `PROVIDER_TIMEOUT_MS` | Таймаут одного запроса к апстриму | 10000 |
| `PROVIDER_MAX_RETRIES` | Попыток повтора при временных сбоях | 3 |
| `PROVIDER_CIRCUIT_THRESHOLD` / `PROVIDER_CIRCUIT_RESET_MS` | Порог размыкания circuit breaker / сброс | 5 / 30000 |
| `CACHE_TTL_ACCOUNT_MS` / `CACHE_TTL_EVENTS_MS` / `CACHE_TTL_DNS_MS` / `CACHE_TTL_NFT_MS` | TTL кэша провайдера | 30s / 60s / 10m / 10m |

Сервер проверяет их при старте и отказывается запускаться при некорректном значении, называя проблемные ключи, но не печатая их. Помните, что лимиты и счётчики блокировок действуют в пределах процесса (см. [LIMITATIONS.ru.md](LIMITATIONS.ru.md)); запускайте один экземпляр приложения, пока не появится общее хранилище лимитов.
